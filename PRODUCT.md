# Product

## Register

brand

## Users

Burned-out solo developers maintaining 10+ OSS repositories with no time to spare.
They are skeptical of hype, allergic to complexity, and have been burned by tools that require constant babysitting.
They arrive with one question: "Will this actually save me time, or create more work?"
Their context is GitHub notifications at 11pm, a backlog of stale issues,
and zero interest in reading a 40-page manual before they decide.

## Product Purpose

Foreman automates the boring parts of OSS repository maintenance — issue triage, labeling, responses —
so maintainers can focus on code.
It acts as a dedicated AI co-maintainer with its own GitHub identity, running always-on in the background.
Success looks like: a maintainer installs Foreman, configures one repo,
and has issues triaged without writing code in under 30 minutes.
The docs site exists to help visitors quickly determine if Foreman is the right tool for them,
then move them toward a walkthrough or deeper feature exploration.

## Brand Personality

Reliable, direct, no-nonsense.
The construction-foreman metaphor is apt: a job-site supervisor who gets things done, doesn't over-explain,
and is still there the next morning.
Not a startup pitch — a piece of infrastructure.

## Anti-references

- Typical AI startup landing pages: gradient blobs, hero metrics, "10x your productivity" copy, glassmorphism cards,
    animated counters
- Generic developer-tool dark modes that look like GitHub Docs with a dark theme swap
- Overly friendly and approachable tones that feel like a SaaS onboarding flow

## Design Principles

1. **Earn trust in the first paragraph.** The user is skeptical.
    Every claim must be concrete and specific — no vague promises about "streamlining workflows."
    Show what Foreman actually does.
2. **Infrastructure confidence.**
    This should feel like a tool that will still be running in three years.
    Solid, durable, unfussy — not trendy.
3. **Complexity hidden, capability shown.**
    Foreman is architecturally complex; the docs should not make it feel that way up front.
    Surface what it does, not how it works, until the reader chooses to go deeper.
4. **Direct over decorative.**
    No padding words, no decorative sections that exist to fill space.
    Every element earns its place.
5. **Construction-grade.**
    The brand has a physical-world metaphor — lean into the solidity of that.
    This is not software-as-art; it is a tool built for the job.
